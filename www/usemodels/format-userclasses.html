Format of user classes

Fully specified form

(anchor for html link to unique-id goes here)
Name: UNIQUE-ID: Short descriptive phrase
Extends: Name/ID of similar user class that is being extended
Summary: A few sentences to sum it up. Try to use words that you might want to search for.
Domain Knowledge: How much does this kind of user know about the application domain and the data being worked with? Rate High, Med, or Low. Or describe.
System Knowledge: How much does this kind of user know about the system being used? Rate High, Med, or Low. Or describe.
Interaction Profile: Rate some of the following High, Med, or Low: Frequency, Intensity, Predictablity, or others as needed.
Key Requirements: List out some of the things that this class of user really needs. Dont make this duplicate the requirements document, just highlight a few to help explain the user class.
Usability Objectives: Rank the following, one per line, or same line if tied:
learnability, rememberability, clarity
efficenty, accuracy, reliability
attractiveness
other:
Notes/Questions:
  • The goal of writing user classes is to prompt and guide careful thought about how they system will be used and how it should be built.
  • In writing or reviewing user classes it is very common for questions or comments to come up, they can be noted here s the same thoughts will be prompted again.
  • Serious issues should also be entered into an issue tracking system and/or trigger the review of associated requirements.

Tips

  • Consider interaction between tigris roles and user classes
  • Consider interaction between tigris user classes and type of site
  • Consider where users come to the site from
  • Consider the user's position in the partitipation pyramid or other community model
  • Consider the knowledge that users have at the start and at each step
  • This format is inspired by the Usage-Centered Design method.
