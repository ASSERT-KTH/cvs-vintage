Format of screen descriptions

Fully specified form

(anchor for html link to unique-id goes here)
Name: UNIQUE-ID: Short descriptive phrase
Purpose: One-sentence description of what this screen is supposed to do.
Contents:
  1. List of abstract widgets on the screen, one per line
  2. Widgets are numbered as a quick way to judge complexity of the screen and model layout at a very high level
  3. Of the form "Name: optional-type descriptive phrase
    • Sub-bullets can have additional comments about their content
    • Conditionals written "If condition: content description"
  4. Boxes or panels can also be listed
    1. Abstract widgets within a box or panel are indicated by indentation
Entrances: How do you get into this screen? links to screens that take you here.
Exits: How do you get out of this screen? links to screens that you can go to next.
Feature(s): Links to the project feature list.
Notes/Questions:
  • The goal of describing screens is to prompt and guide careful thought about how they system will be used and how it should be built.
  • In writing or reviewing screens it is very common for questions or comments to come up, they can be noted here so the same thoughts will be prompted again.
  • Serious issues should also be entered into an issue tracking system and/or trigger the review of associated requirements.
  • Also, note any properties of the screen that are relevant, but not called-out in the form.
Layouts: Notes on final screen visual layout, or links to wireframes or mockups. If there are existing screens in a previous version of the system, or screens in some reference or competitive product, you can refer to them at first. This is last because it is to be done once the other aspects of each screen are thought through.

Tips

  • Keep widgets abstract, dont choose concrete widgets too soon.
  • First content, then layout, then iterate.
  • Compare abstract widget list to use case step details.
  • Compare entrances and exits to use case step sequences.
  • Consider screen complexity
  • Consider how screens scale to large datasets
  • Consider information update rate if applicable
  • Consider permissions as conditions on content item display
  • Consider on-screen and on-line help
  • This format is inspired by the Usage-Centered Design method.
