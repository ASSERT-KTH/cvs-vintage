Name: UC-08: Assigning an issue to specific developer
Extends: None
Purpose: If an issue has been entered and was not assigned to a specific developer at that time, the component owner must assign it to a specific developer.
Feature(s): FeatureName
Direct Actors: CompOwner: Component Owner (High Priority, High Frequency)
Assignee: Typically Developer (High Priority, High Frequency)
Scarab: The scarab system.
Stakeholders: Reporter: Wants to know when work on reported issue has begun, when it might be resolved.
Assignee: needs to know about issues assigned to him/her.
Assignee: may be a person or a shared work queue.
Other project members: want to have visibility into project activity and progress.
Pre-conditions: Logged in
Project set up
Existing issue assigned to component owner by default
Post-conditions: Issue assigned to assignee, and intersted parties notified and ready to take action. Next step, assignee accepts issue and begins work.
Data sets: Assignee name
Main Success Scenario:
  1. CompOwner: visit MyIssues or other list of issues assigned to his/her component
  2. CompOwner: select an issue from list
  3. Scarab: display IssueView
  4. CompOwner: review issue and decide on assignee
  5. CompOwner: enter new assignee and comment and hit submit
  6. [validation] Scarab: validates assignee and comment
  7. Scarab: record change
  8. Scarab: send email to old and new assignees and other users who expressed interest in issue
  9. Scarab: display change comfimation, email confirmation, and IssueView for next issue in list, or IssueList if last
Alternative Scenario Extensions:
  • [validation] if validation fails:
    • Scarab: display IssueView w/ error message and field data as entered
Notes/Questions:
  • In bugzilla, anyone can reassign any issue to anyone. In Issuezilla, permissions are limited somewhat and assignees are limited to project members. What enterprise-level requirements are there on who can assign what to whom in scarab?
  • Workload and relevant knowledge are factors in making good assignments.
  • Estimated effort to resolve?
  • Is this really more of a 'triage incoming issues' use case?
  • Make this extend a general editing use case.


Name: UC-09: Changing an issue
Extends: None
Purpose: General use case for any kind of issue field change.
Feature(s): FeatureName
Direct Actors: User: Any Role (High Priority, High Frequency)
Scarab: The scarab system.
Stakeholders: None
Pre-conditions: Logged in
Project set up
Existing issue
Post-conditions: Change is validated and recorded, user sees confirmation, notificiations emailed to interested parties.
Data sets: Query terms.
Main Success Scenario:
  1. User: visit IssuesView.
  2. User: review fields in issue
  3. [change] User: change field values and enter comment
  4. [validation] Scarab: validates field values
  5. Scarab: record change
  6. Scarab: send email to old and new assignees and other users who expressed interest in issue
  7. Scarab: display change comfimation, email confirmation, and IssueView for next issue in list, or IssueList if last
Alternative Scenario Extensions:
  • [validation] if validation fails:
    • Scarab: display ProjectIssues with error message and field data as entered
Notes/Questions:
  • Future feature: This is where query relaxation could be useful to offer users results that were "close" to what they asked for in addition to the ones that exactly match.
  • Future feature: result column sorting, custom columns, etc.
