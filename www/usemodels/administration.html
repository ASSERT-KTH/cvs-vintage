Administration Use Cases

Name: UC-23: User Defines Issue Status
Extends: None
Purpose: Allow the DomainAdmin or ProjectOwner to define the issue status across projects (domain) or within a specific project
Feature(s): FeatureName
Direct Actors:
  • ProjectOwner:
  • DomainAdmin
  • Other Users on the Project
  • Scarab: The scarab system.
Stakeholders: All users on the project and/or withing the domain
Prereq:
  • Logged In as DomainAdmin or ProjectOwner
  • Can be done prior to project creation or after the fact
Results: Issue Status is defined as per user input
Data sets:
Notes/Questions:
  • Should this be only at a domain level or does it make sense to have the ability to make this project specific?
  • Possible conflicts can occur if this allowed at the project level (i.e. attempting to export or import databases because of different nomenclature)
  • Educating point to DomainAdmin and ProjectOwner in case of different naming conventions
Main Success Scenario:
  1. DomainAdmin or ProjectOwner: navigates to ProjectIssues page
  2. DomainAdmin or ProjectOwner: clicks on the Options tab , then the Customization tab/link, and finally the Define Issue Status link.
  3. Scarab: take user to the DefineIssueStatus page
  4. DomainAdmin or ProjectOwner: has three options:
    1. If Add:
      1. User enters name of new state (free text) and clicks Add. Goto Step 5
    2. If Edit:
      1. Scarab: display a list of predefined Status to change
      2. User: makes changes and clicks Update. Goto Step 5
    3. If Delete:
      1. Scarab: display a list of predefined Status to delete
      2. User: selects one or several Status and clicks Delete
      3. Scarab: display confirmation window or message
      4. User: confirms and clicks Delete.
  5. Scarab: display DefineIssueStatus page with IssueChangedConfirmation
Alternative Scenario Extensions:
  • None

Name: UC-29: User Imports an XML File (Database) into a Project
Extends: None
Purpose: Allows user to import an XML file (database) into a project. The user can easily transfer an issue database across projects in an XML format.
Feature(s): FeatureName
Direct Actors: CompOwner: Component Owner (High Priority, High Frequency)
Other Users on the Project
Scarab: The scarab system.
Stakeholders: CompOwner:
Other project members: The ability to view/download the attached file(s)
Prereq: Logged In
Project set up
User has access to the project
User has the necessary DTD
Results: Issue database are imported to selected projects
Data sets:
Notes/Questions:
  • Does the user have access to the project?
  • Should Scarab allow a partial transfer in cases of bad data? Should it be all or nothing?
  • Should Scarab check for incremental updates or just allow the transfer of the db without checking?
  • No anonymous users can import XML files to an issue
  • At what point do users have to log in?
  • Make sure user has the DTD.
  • Project Owner or Domain Admin role permission only?
Main Success Scenario:
  1. ProjectOwner: navigates to ProjectIssues page
  2. ProjectOwner: clicks on the Options tab and clicks the Import link
  3. Scarab: display a window which allows the user to either enter the path/name of the XML file or to browse their local environment to select the XML file(s) they want to atach (Note: Browsing allows the ability to select several files at once)
  4. ProjectOwner: enters the name or selects the file(s) from their local environment.
  5. ProjectOwner: provides a brief description of the file(s) and clicks Import
  6. Scarab: checks format vs DTD
  7. Scarab: writes good data (see notes/questions on all or partial) to database
  8. Scarab: creates a log file with data summary and/or error summary
  9. Scarab: display ProjectIssues page with IssueChangedConfirmation
Alternative Scenario Extensions:
  • None
