Use Cases

  • UC-01 - Entering an Issue
Name: UC-01 - Entering an Issue
Extends: None
Description: User enters an issue, duplicates are avoided if possible. The static fields of an issue are entered first along with a short description. This is used to look for duplicate issues. If it is a duplicate, the currently selected issue gains a vote. If a new issue is entered, it is well specified and actionable.
Feature(s): None
Direct Actors:
  • User : Any Role (High Priority, High Frequency)
  • Scarab : The scarab system
Stakeholders:
  • QA Manager : wants to avoid duplicate issues.
  • QA Manager : wants enough info to assign issue to proper developer, deadline.
  • QA Manager : wants well-formed, well-written issue reports.
  • Assigned Developer : wants detailed description, steps to reproduce.
PreRequirements:
  • Project set up
  • User knows which project to report the issue against
Result: New issue is entered, or matching existing issue has vote increased.
DataSets:
  • Quick fields
  • Yes/No on duplicate candidates
  • Detailed fields
  • Field values based on user profile: e.g., reporter
Q/A:
  • Does the user have limited votes?
  • Do the forms help users enter good issue reports?
  • At what point do users have to log in?
    • Users need to log in before making modifications to the system. In other words, in order for the system to track modifications a user makes, the user needs to be logged in.
Notes:
  • No anonymous users can post issues
  • Admin use case to to put which fields in the NewIssueBox
Senario:
  1. User: visit ProjectIssues for a given project
  2. User: log in as non-guest if not already
  3. User: enter fields in NewIssueBox, hit submit
    • choose type of issue
    • enter summary
    • enter platform, os
    • enter severity, priority (role specific)
  4. Scarab: checks for duplicates
  5. If duplicates found:
    1. Scarab: display Deduplicate Page
    2. User: review list of duplicates
    3. If dup matches user's issue:
      • User: vote for matching issue
    4. If no match:
      • User: indicate no match and show the next found potential duplicate issue. If no more duplicate issues, then show NewIssueDetails page
  6. Scarab: display NewIssueDetails page
  7. User: enter fields and hit submit
  8. Scarab: validate data and show next screen or bounce back to current screen with previous information populated again.
  9. Scarab: store values in database
  10. Scarab: display NewIssueAddendum page with IssueChangedConfirmation
  11. User: options to attach a file to issue. Scarab: needs to track and deal with this properly.
  12. Scarab: display ProjectIssues page with IssueChangedConfirmation
