Browsing and Searching Use Cases

Name: UC-02: Search for issue by ID
Extends: None
Purpose: User enters issue number and sees the specified issue or an appropriate message.
Feature(s): FeatureName
Direct Actors: User: Any Role (High Priority, High Frequency)
Scarab: The scarab system.
Stakeholders: Fellow Project Participant: wants to be able to specify issue unamgiously via an easy-to-type-or-say string. This is often done via email, in code comments, in chat, or verbally on phone or in meetings. Challenge is that unambiguous specification is at odds with brevity. Assumes project name or URL as shared knowledge?
Pre-conditions: Logged in
Project set up
Existing issue
User knows issue number
Post-conditions: Specified issue is displayed or error message.
Data sets: Issue number.
Main Success Scenario:
  1. User: visit ProjectIssues, or any Scarab page in project context.
  2. User: Enter issue ID into small field and hit button
  3. [validation] Scarab: validates issue number
  4. Scarab: display IssueView
Alternative Scenario Extensions:
  • [validation] if validation fails, Scarab displays InvalidIssueNumber page
Notes/Questions:
  • If all issues are in project context, is there a quick way to jump to issue in another project?
  • How can this functionality be put on tigris StartPage?


Name: UC-03: Quick search for issues
Extends: None
Purpose: User enters a few terms and sees a list of issues that are likely to be relevant. Next, the user can drill into a specific issue from the list.
Feature(s): FeatureName
Direct Actors: User: Any Role (High Priority, High Frequency)
Scarab: The scarab system.
Stakeholders: None
Pre-conditions: Logged in
Project set up
Existing issue(s)
User knows some search terms
Post-conditions: List of issues matching criteria are displayed, or go direcly to issue if only one match.
Data sets: Query terms.
Main Success Scenario:
  1. User: visit ProjectIssues.
  2. User: Enter search critia into field and hits 'search' button
  3. [validation] Scarab: validates search criteria
  4. Scarab: searches for matching issues
  5. if results.size( ) == 0:
    • Scarab: display message on ProjectIssues with field data as entered
    • User: Modify search criteria and repeat
  6. if results.size( ) == 1:
    • Scarab: display IssueView
  7. if results.size( ) > 1:
    • Scarab: display IssueList
    • User: survey resulting list
    • User: select one of the presented issues
Alternative Scenario Extensions:
  • [validation] if validation fails:
    • Scarab: display ProjectIssues with error message and field data as entered
Notes/Questions:
  • Future feature: This is where query relaxation could be useful to offer users results that were "close" to what they asked for in addition to the ones that exactly match.
  • Future feature: result column sorting, custom columns, etc.
