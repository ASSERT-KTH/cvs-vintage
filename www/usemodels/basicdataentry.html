Basic Data Entry Use Cases

Name: UC-01: Creating an New Issue
Extends: None
Purpose: User enters an issue, duplicates are avoided if possible. The static fields of an issue are entered first along with a short description. This is used to look for duplicate issues. If it is a duplicate, the currently selected issue gains a vote. If a new issue is entered, it is well specified and actionable.
Feature(s): FeatureName
Direct Actors: User: Any Role (High Priority, High Frequency)
Scarab: The scarab system.
Stakeholders: QA Mangager: wants to avoid duplicate issues.
QA Mangager: wants enough info to assign issue to proper developer, deadline.
QA Mangager: wants well-formed, well-written issue reports.
Assigned Developer: wants detailed description, steps to reproduce.
Prereq: Project set up
User knows which project to report the issue against
Results: New issue is entered, or matching existing issue has vote increased.
Data sets: Quick fields
Yes/No on duplicate candidates
Detailed fields
Field values based on user profile: e.g., reporter
Notes/Questions: Does the user have limited votes?
Do the forms help users enter good issue reports?
No anonymous users can post issues
At what point do users have to log in? Users need to log in before making modifications to the system. In other words, in order for the system to track modifications a user makes, the user needs to be logged in.
Admin use case to to put which fields in the NewIssueBox
Main Success Scenario:
  1. User: visit ProjectIssues for a given project
  2. User: log in as non-guest if not already
  3. User: enter fields in NewIssueBox, hit submit
    • choose type of issue
    • enter summary
    • enter platform, os
    • enter severity, priority (role specific)
  4. Scarab: checks for duplicates
  6. If duplicates found:
    1. Scarab: display Deduplicate Page
    2. User: review list of duplicates
    3. If dup matches user's issue:
      • User: vote for matching issue
    4. If no match:
      • User: indicate no match and show the next found potential duplicate issue. If no more duplicate issues, then show NewIssueDetails page
  7. Scarab: display NewIssueDetails page
  8. User: enter fields and hit submit
  9. Scarab: validate data and show next screen or bounce back to current screen with previous information populated again.
  10. Scarab: store values in database
  11. Scarab: display NewIssueAddendum page with IssueChangedConfirmation
  12. User: options to attach a file to issue. Scarab: needs to track and deal with this properly.
  13. Scarab: display ProjectIssues page with IssueChangedConfirmation
Alternative Scenario Extensions:
  • None
