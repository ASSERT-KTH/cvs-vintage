Format of use cases

Fully specified form

Name: UNIQUE-ID: Short descriptive phrase
Extends: Name/ID of similar use case that is being extended
Purpose: A few sentences to sum it up. Try to use words that you might want to search for.
Feature(s): Links to the project feature list.
Direct Actors: A list of "actor-id : user-class (priority, frequency)" one per line.
actor-id is a label that will be used to identify these actors in the steps.
user-class is the kind of user or system that they are. User classes are listed and described in a separate document.
priority is how imporant this use case is to this user, not all users will put emphasis on the same use cases.
frequencey is how often the user works through this use case.
"Direct actors" are basically the people at the keyboard and the systems they use. Often if there is just one, we can name it "User". There can be several direct actors in use cases where people collaborate closely or where one person needs to use several systems to get the task done.
Stakeholders: A list of "User-class: description of involvment".
Stakeholders are other people who care about how the system are used, but are not actually sitting at the keyboard during this use case.
Pre-conditions: Conditions that we assume are met before this use case is started.
Post-conditions: Short sentence or two about the situtation after the use case has been done.
Data sets: If there is something to say about the data entered or seen in the steps of the use case, it can be specified here to avoid cluttering the descriptions of the steps.
Main Success Scenario:
  1. Steps leading to the successful completion of the use case
  2. Each takes the form "[label] actor-id: short pharase in active voice"
  3. frequently, steps will alternate between user intention and system reaction, but obvious system reactions can be skipped, and detail system reactions can be shown as multiple steps.
  4. If actions refer to interaction contexts (pages or screens), they should be linked to descripttions of those.
  5. Conditionals are written "If condition:" and then indented bullets list alternatives.
  6. Very short conditions and actions can be written on one line.
  7. Sometimes a step can have unordered bullets under it to list small substeps where order is not important or determined. For example when filling in a form, you might bullet-list the fields requred for this use case.
  8. Labels are written: "[label-name]" and can be applied to any step. Even though steps are numbered, the numbers can change and shold not be used for reference.
  9. Loops are written as "While Condition:" and then indented steps.
  10. User progress through the Main Success Scenario should be fairly straight-line. If you find you are writing a lot of conditionals, you probably need to break up the use case into more specific ones, or list alternatives in the next section.
Alternative Scenario Extensions:
  • Each item here should make reference to one of the labeled steps above and provide an alternative or optional additional step.
  • One of the biggest uses for this is specifying error conditions.
  • For example: "[validation] if validation fails, Scarab displays InvalidIssueNumber page"
Notes/Questions:
  • The goal of writing use cases is to prompt and guide careful thought about how they system will be used and how it should be built.
  • In writing or reviewing use cases it is very common for questions or comments to come up, they can be noted here s the same thoughts will be prompted again.
  • Serious issues should also be entered into an issue tracking system and/or trigger the review of associated requirements.


Short form

(anchor for html link to unique-id goes here)
Name: UNIQUE-ID: Short descriptive phrase
Extends: Name/ID of similar use case that is being extended
Purpose: A few sentences to sum it up. Try to use words that you might want to search for. You can also include other sections if they are key to the use case. The point is to get all required use cases into some document at some level of specificity, so that they are not lost and so that the level of completeness of the overall requirements specification can be measured.


Tips

  • Consider interaction between tigris roles and user classes
  • Consider interaction between tigris user classes and type of site
  • Consider interaction between user classes and permissions
  • Consider the knowledge that users have at the start and at each step
  • For each prerequiste fact, is there any way the use case could changed to eliminate or delay the need for that assumption?
  • This format is inspired by the Usage-Centered Design method.
